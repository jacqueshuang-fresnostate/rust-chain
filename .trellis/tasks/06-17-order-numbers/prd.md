# 订单展示改为业务订单号

## Goal

用户可见的订单号不再直接展示数据库主键 `id`，统一展示独立业务订单号 `order_no`；内部接口路径、审批、取消、还款等操作仍可继续使用主键 `id`，避免破坏现有操作链路。

## What I Already Know

- 用户明确要求：所有显示订单号的地方，不要使用 `id` 作为订单号，而是生成订单号。
- 快速充值已有平台订单号 `order_id`，不需要改成主键。
- PC 理财订单页面当前直接显示 `order.id` 为“订单号”。
- 后台资源页中闪兑订单、理财申购、新币上市认购等仍有 `id` 作为“订单ID/申购ID”展示。
- 现货、秒合约、贷款等业务内部操作 path 仍依赖主键 `id`，不能直接替换为 `order_no`。

## Assumptions

- 本次只处理用户可见的订单编号展示，不改变内部主键和现有详情/操作接口路径。
- `order_no` 可以由业务前缀、创建日期和主键组合生成，满足不显示裸 `id` 的需求，并兼容历史数据。
- 快速充值的 `order_id` 已是外部平台订单号，继续保持。

## Requirements

- 展示层统一生成业务订单号；如果后端订单响应已经提供 `order_no`，则优先展示后端字段。
- PC 展示订单号的位置使用 `orderNo` / `order_no`，不能直接显示 `id`。
- 后台订单类表格使用业务订单号展示“订单号/申购号”，内部 `id` 保留在 record 里用于行级操作。
- 不修改旧迁移，避免已应用迁移 checksum 问题。

## Acceptance Criteria

- [ ] PC 理财订单页面显示 `orderNo` / `order_no`，不显示 `order.id` 作为订单号。
- [ ] 后台闪兑订单、理财申购、新币上市认购等订单列显示 `order_no`。
- [ ] 后端相关响应返回稳定的 `order_no` 字段。
- [ ] 现有取消、审批、详情等操作继续使用 `id`，行为不变。
- [ ] 相关后端测试、PC 静态测试、后台资源配置测试通过。

## Out Of Scope

- 不把所有内部 `id` 路由替换为 `order_no`。
- 不处理非订单含义的 ID，例如用户ID、资产ID、仓位ID、流水ID。
- 不改快速充值对 GMPay 使用的 `order_id` 合同。

## Technical Notes

- 新增迁移应使用新的版本号，不修改 0042/0052/0054/0071 等已应用迁移。
- 本次 MVP 使用前端工具按业务前缀、日期和非裸露 ID token 生成业务编号，兼容历史数据；后端未来可直接返回 `order_no` 覆盖展示。
