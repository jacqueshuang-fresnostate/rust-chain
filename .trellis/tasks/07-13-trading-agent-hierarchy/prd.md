# 打通现货、杠杆、秒合约与多级代理后端逻辑

## 目标

在不推翻现有 DDD 模块和钱包账本模型的前提下，补齐现货、杠杆、秒合约的关键闭环，并把当前单层代理归属升级为“后台超级管理员 > 总代理 > 二级代理 > 三级代理”的可查询组织树。

## 已知现状

- 现货已有市价、限价、止损限价、冻结、成交、撤单、行情触发和幂等逻辑，但缺少服务端批量撤单入口。
- 杠杆已有单仓/全部平仓、单个/全部撤销、现货与杠杆钱包划转、默认杠杆和保证金模式设置；现有测试未覆盖这些新增动作的完整资金与权限链。
- 秒合约已有产品周期、扣款、后台/自动结算和幂等逻辑，但无 Redis 或无有效开仓行情时仍可扣款创建 `entry_price = NULL` 的订单，自动结算会永久重试。
- 当前 `agents.level` 只是自由整数；代理之间没有父子关系。`user_referrals.root_agent_id` 实际表示用户直接归属的代理，代理查询均使用等值过滤，只能看到本代理直接团队。
- 后台管理员保持独立的 Admin Token/RBAC，不需要伪造一条代理数据；在代理业务语义中视为虚拟 0 级超级代理并拥有全局视图。

## 功能范围

### 1. 代理组织树

- `agents` 增加父代理、根代理和物化路径字段，保留并收紧 `level`：
  - 总代理：`level = 1`、无父代理。
  - 二级代理：`level = 2`、父代理必须为总代理。
  - 三级代理：`level = 3`、父代理必须为二级代理。
  - 禁止创建第四级代理。
- 迁移现有代理为独立总代理，保证历史数据继续可用。
- 后台创建代理时接收可选 `parent_agent_id`，等级、根代理和路径由服务端推导；若旧请求携带 `level`，必须与推导结果一致。
- 后台代理列表/详情返回父级、根级、路径和团队统计，并支持按父级、根级、等级过滤。
- 后台 `GET /agents/:id/users` 返回该代理整棵子树中的用户，而不是只查等值 `root_agent_id`。
- 代理端 `/users`、`/dashboard` 的团队人数、`/convert/stats`、`/team-tree` 按当前代理子树计算；不能看到父级、兄弟代理或其他树数据。
- 代理端 `/commissions` 只返回当前代理本人名下且来源用户位于其子树内的佣金记录；组织可见性不改变佣金资金归属，也不允许查看下级代理的结算记录。
- 代理端提供下级代理列表；用户响应包含其直接归属代理信息，便于后台正确展示组织关系。
- 任何祖先代理停用后，其所有下级代理均不能登录或访问代理接口；恢复祖先后，下级自身仍为 active 时可恢复访问。
- 邀请码继续只由所属代理管理；用户邀请链保持归属直接代理，不重写历史 `user_referrals.path`。

### 1.1 代理归属与用户邀请链

- 系统同时保留两种关系，但两者不是互相覆盖的两棵树：
  - 代理组织树表示“用户归属哪一家代理公司”，通过直属归属代理及其 `agents.path` 决定上级可见范围。
  - 用户邀请链表示“具体是谁介绍该用户”，通过 `direct_inviter_type/direct_inviter_id` 和用户路径记录。
- 代理邀请用户 A 时，A 的直属邀请人为该代理，且 A 归属该代理公司。
- 用户 A 邀请用户 B 时，B 的直属邀请人为 A，但 B 必须继承 A 的直属归属代理；A 后续邀请的所有层级用户均归属同一家公司。
- 直属归属代理及其所有上级代理可以看到 A、B 和后续用户；兄弟代理、其他代理树和无关用户不能看到。
- 上级代理链任一节点停用后，该公司下用户的个人邀请码也不得继续注册或绑定新用户。
- 代理端和后台团队用户响应同时返回明确的 `owner_agent_id` 与 `direct_inviter_type/direct_inviter_id`；保留历史 `root_agent_id` 作为直属归属代理的兼容字段。

### 2. 现货交易

- 新增用户侧服务端批量撤单接口，可按可选交易对过滤。
- 每个订单复用现有单单撤销、解冻和事件逻辑，重复请求不得重复解冻。
- 市价单只允许使用 60 秒内有效服务端行情成交；客户端参考价只用于滑点保护，行情缺失不得成交或冻结资产。
- 用户成交列表必须按当前用户买方/卖方身份过滤，不能泄露全市场用户成交记录。
- 批量撤单逐项继续执行并返回失败项，单项失败不得阻断后续订单。
- 保持市价、限价、止损限价和后台人工撮合现有接口兼容。

### 3. 杠杆交易

- 对全部平仓、单个/全部撤销、现货/杠杆钱包双向划转、修改默认杠杆和切换默认保证金模式补齐后端集成测试。
- 增加读取用户某产品杠杆设置的接口，保证保存后的默认设置可被客户端恢复。
- 新开仓必须取得有效行情；历史遗留的无开仓价仓位仍可撤销，已成交仓位必须平仓。
- 平仓、撤销和爆仓资金按仓位 `wallet_scope` 原路返回。
- 划转增加服务端幂等记录和资产精度校验；同键同参数安全重放，同键异参冲突。
- 当前 `cross` 没有账户级共享风控语义，用户设置和开仓必须明确拒绝，不能把逐仓算法伪装成全仓。
- 批量平仓/撤单不设固定 100 条静默上限，并逐项继续执行、汇总失败。
- 资金操作继续保证余额、流水和业务状态同事务提交；批量操作复用单笔幂等逻辑。

### 4. 秒合约

- 开仓必须取得 60 秒内、价格为正的 Redis 行情；行情缺失、过期或 Redis 未配置时在扣款和写订单前拒绝。
- 开仓要求产品、交易对、质押资产和交易对两侧资产全部启用，并校验质押金额资产精度。
- 手工和自动结算统一按质押资产精度向下截断派奖金额。
- 保持订单、钱包扣款、流水、周期快照与幂等键同事务提交。
- 自动结算继续持久化结算价并幂等派奖；补充缺失行情不能开仓的测试。

### 5. 多业务多级差额返佣

- 返佣规则支持 `convert`、`prediction`、`spot`、`margin`、`seconds_contract` 五种业务；不存在启用规则的业务不产生返佣。
- 每条代理规则的 `commission_rate` 表示该代理在对应业务上的累计返佣比例，不是每一级都独立叠加的比例。
- 用户产生可返佣业务时，从直属归属代理向总代理依次计算差额：直属代理获得其累计比例，上一层只获得“自身累计比例 - 下级链已分配最高比例”的正差额。
- 示例：三级代理 5%、二级代理 8%、总代理 10%，实际返佣依次为 5%、3%、2%；若某层没有启用规则则跳过，由更上层按累计比例补足。
- 任何层级配置倒挂时只分配正差额，总返佣比例不得超过本次祖先链启用规则中的最高累计比例。
- 各业务返佣基数和结算资产：闪兑使用源资产金额与源资产；竞猜使用质押金额与质押资产；现货买方使用成交计价金额与计价资产、卖方使用成交数量与基础资产；杠杆使用开仓保证金金额与保证金资产；秒合约使用质押金额与质押资产。
- 返佣记录必须在对应成交/开仓资金事务中写入，使用代理、来源类型和来源 ID 保证逐级记录幂等，并快照本次实际差额比例。
- 返佣金额必须按结算资产 `precision_scale` 向零截断；截断后为零时不创建无效记录。
- 代理端仍只查看自己名下的返佣记录；上级团队可见性不能泄露下级代理的返佣记录。
- 后台返佣规则页面必须可以创建、筛选和维护全部五种业务规则，并展示返佣记录的实际差额比例。

## 权限与数据边界

- Admin 作用域：可查看和管理全部代理树及全部用户。
- Agent 作用域：根边界为当前登录代理的 `agents.path`；只允许访问该路径本身及其后代路径。
- User 作用域：交易接口只能操作当前 Token 对应用户的钱包、订单和仓位。
- 所有钱包变更必须有同事务流水；代理树查询不得依赖客户端传入的代理范围。

## 兼容性

- 现有代理全部回填为一级总代理。
- 保留 `user_referrals.root_agent_id` 字段和既有邀请数据；它继续表示用户的直接归属代理。
- 新增响应字段 `owner_agent_id` 明确表达直属归属代理，不改变历史 `root_agent_id` 的存储语义。
- 现有 API 字段不删除，只新增层级字段和批量/设置读取接口。
- 现有 PC、Mobile 调用的交易路由不改名。

## 非目标

- 本任务不新增完整的杠杆限价委托簿或撮合前订单表。
- 本任务不修改后台或手机端页面。
- 本任务不允许代理自行创建下级代理；本轮由超级管理员创建并指定父级。

## Definition of Done

- 迁移、领域规则、仓储查询、应用用例、路由 DTO 和测试同步完成。
- 风险敏感逻辑保留简洁中文注释，测试代码只放在 `tests/`。
- 更新后端规范与 `docs/superpowers/PROGRESS.md`。

## 技术说明

- 代理树采用物化路径，子树过滤使用 `path = ? OR path LIKE CONCAT(?, '/%')`，避免递归 CTE 分散到多条查询。
- 用户保持直接归属代理，通过“直接代理属于当前代理子树”实现向上可见。
- 详细现状审计见 [`research/backend-gap-audit.md`](research/backend-gap-audit.md)。
