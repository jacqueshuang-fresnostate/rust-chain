# 01. 项目总览与整体架构设计

## 1. 文档目标

本文档定义区块链交易所平台一期 MVP 的整体架构。平台采用混合模式：主流币交易对接入 Bitget 与 HTX 行情，新币使用内部交易对与策略行情。平台发行的币在全部设计文档中统一称为新币。

## 2. 平台模式

平台采用混合交易所模式：

- 主流币交易对展示 Bitget / HTX 外部行情。
- 新币交易对使用内部策略行情。
- 外部行情、内部撮合、策略行情在数据模型、权限和风控上隔离。
- 后台行情策略仅允许作用于新币或内部测试交易对。
- 用户资产以 available、frozen、locked 三类余额管理，所有变动必须有流水。

## 3. 一期 MVP 范围

一期实现：

- 用户与认证
- 资产账户与资产流水
- Bitget / HTX 行情适配
- K 线查询与聚合
- 现货交易
- 闪兑
- 新币发行、申购、派发、上市、认购与解禁
- 新币策略行情与 K 线停机补偿
- 平台管理员后台
- 代理、邀请、代理后台
- 风控、审计、权限隔离

二期预留：

- 秒合约
- 杠杆交易
- 理财产品
- 做市机器人
- 链上充值提现
- 多级返佣结算

| 层级 | 方案 |

| Web 框架 | Axum 优先，Actix 作为可选替代 |
| 交易数据库 | MySQL |
| 行情文档库 | MongoDB |
| 缓存 | Redis |
| 消息队列 | RabbitMQ |
| 行情源 | Bitget + HTX |
| 部署形态 | 模块化单体服务 + MySQL + MongoDB + Redis + RabbitMQ |

## 5. 架构原则

一期采用模块化单体 MVP：

- 一个 Rust 后端工程，按领域模块拆分。
- 用户、资产、行情、交易、闪兑、后台、新币、代理模块保持清晰边界。
- 撮合引擎、行情适配层、策略行情引擎、异步任务按可拆服务边界设计。
- MySQL 承载核心业务状态，MongoDB 承载历史行情，Redis 承载实时缓存和锁，RabbitMQ 承载异步事件。
- 资产变更、订单状态、认购、解禁、闪兑必须以 MySQL 事务和幂等键保证一致性。

## 8. 核心数据流

### 8.1 用户请求链路

1. 客户端请求 REST API 或 WebSocket。
2. 接入层完成参数校验、限流和认证。
3. 权限层按 token scope 区分 user、admin、agent。
4. 业务域层执行领域规则。
5. 数据访问层写入 MySQL、Redis 或 MongoDB。
6. 必要时发布 RabbitMQ 事件。
7. WebSocket 推送订单、资产、行情或通知变化。

### 8.2 行情链路

1. Market Adapter 连接 Bitget / HTX REST 与 WebSocket。
2. 外部 ticker、depth、trade、kline 标准化为 MarketEvent。
3. 最新 ticker、盘口、短周期 K 线写入 Redis。
4. 标准化行情事件发布到 RabbitMQ。
5. Kline Aggregator 聚合多周期 K 线。
6. 历史 K 线和策略行情结果写入 MongoDB 按交易对拆分的 Collection。

### 8.3 现货交易链路

1. 用户提交限价单或市价单。
2. API 校验登录态、交易对状态、价格精度、数量精度、最小下单额和风控。
3. 资产模块冻结对应资产并写资产流水。
4. 订单进入撮合模块。
5. 撮合成功后生成成交记录。
6. 资产模块按成交结果解冻、扣减、入账并写流水。
7. 发布 order.matched 与 wallet.balance_changed 事件。
8. WebSocket 推送订单状态、成交和资产变化。

### 8.4 新币生命周期链路

1. 后台创建新币、发行信息、解禁规则和交易对。
2. preheat 阶段只能查看，不能申购或购买。
3. subscription 阶段允许申购，新币未到账。
4. distribution 阶段执行派发，新币进入 available 或 locked。
5. listed 阶段启用交易对，K 线开始展示，用户可现货交易或通过认购入口购买。
6. 认购所得继续继承币种解禁规则。
7. 到期后未启用矿工费可自动释放；启用矿工费时必须先支付矿工费再释放。

### 8.5 闪兑链路

1. 用户请求闪兑报价。
2. 主流币读取外部行情和平台点差，新币读取后台配置汇率。
3. 报价写入 Redis 并设置短 TTL。
4. 用户确认报价。
5. MySQL 事务内扣减 from_asset available、增加 to_asset available、写双边流水和闪兑订单。
6. 发布 convert.completed 和 wallet.balance_changed 事件。

## 9. 新币锁定资产模型

新币锁定资产分为两层：

| 层级 | 作用 |
|---|---|
| wallet_accounts.locked | 用户某币种锁定总额，用于资产展示和可用性校验 |
| asset_lock_positions | 锁定仓位或锁定订单，用于记录解禁时间、剩余锁定量和释放状态 |
| asset_lock_position_sources | 锁定仓位来源明细，用于追踪申购获配、派发、认购等来源 |

锁定聚合规则：

| 解禁类型 | 锁定存在方式 | 聚合规则 |
|---|---|---|
| immediate_on_listing | 上市时间作为统一解禁点 | 同一用户、同一币种、同一 listed_at 可聚合为一个锁定仓位；到上市时间释放 |
| fixed_time | 固定日期解禁 | 同一用户、同一币种、同一 unlock_at 聚合为一个锁定仓位；申购和认购都会并入该仓位 |
| relative_period | 按订单时间计算解禁 | 每笔申购、派发或认购单独形成锁定订单，不聚合；每笔订单有独立 unlock_at |

设计约束：

- `wallet_accounts.locked` 只保存汇总值，不作为解禁明细来源。
- 固定日期解禁可以聚合，减少用户锁定记录数量。
- 相对周期解禁不能聚合，因为每笔订单的起算时间不同。
- 每次申购获配、认购、锁定、解禁、矿工费支付都必须写 `wallet_ledger`。
- 聚合锁定仓位仍需保留来源记录，确保审计、退款、修正和争议处理可追溯。

## 10. 存储职责

| 数据类型 | 存储 | 说明 |
|---|---|---|
| 用户、资产、订单、闪兑、后台配置、代理关系 | MySQL | 强事务业务数据 |

## 11. 部署拓扑

一期部署为模块化单体：

- `exchange-api`：一个 Rust 服务进程，内含 REST API、WebSocket、领域服务、撮合、行情、策略、任务 worker。
- `MySQL`：核心交易业务数据。
- `MongoDB`：历史行情和 K 线。
- `Redis`：实时缓存、报价 TTL、限流、分布式锁。
- `RabbitMQ`：内部异步事件。
- `Bitget / HTX`：外部行情源。

后续高并发阶段可拆分为：API Gateway、Matching Service、Market Service、Wallet Service、Risk Service、Admin Service。

## 12. 一致性与幂等边界

| 场景 | 一致性规则 |
|---|---|
| 下单冻结 | MySQL 事务内校验余额、冻结资产、写订单和流水 |
| 成交结算 | 成交、解冻、扣减、入账、流水必须同事务或具备补偿机制 |
| 撤单解冻 | 同一订单只能撤单一次，解冻必须幂等 |
| 新币派发 | subscription_id 或 distribution_id 作为幂等键 |
| 上市后认购 | purchase_order_id 作为幂等键，认购资产扣减与新币入账同事务 |
| 上市即解禁锁定 | immediate_on_listing 使用 user_id + asset_id + listed_at 作为上市前锁定 merge key；上市后认购直接进入 available |
| 锁定仓位合并 | fixed_time 使用 user_id + asset_id + unlock_at 作为 merge key |
| 相对周期锁定 | relative_period 使用 source_type + source_id 作为唯一来源键 |
| 解禁释放 | lock_position_id 或 unlock_record_id 作为幂等键 |
| 解禁矿工费 | fee payment 与 release 分步幂等，未支付不能 release |
| 闪兑确认 | quote_id 只能成交一次，报价过期拒绝 |
| 异步事件 | 事件携带 idempotency_key，消费者去重 |

## 13. 全局时间字段边界

- 项目所有涉及时间的对外字段必须统一使用时间戳语义，默认采用 Unix milliseconds 数字。
- REST API 请求/响应、WebSocket 消息、RabbitMQ 事件 payload、Redis 缓存 payload 和 MongoDB 行情文档都必须遵守该边界，不输出本地化时间字符串。
- Rust 服务内部可继续使用 `DateTime<Utc>` 做运算，MySQL 可继续使用 `TIMESTAMP(6)` 做持久化；进入或离开外部边界时必须转换为 Unix milliseconds。
- 新增模块、字段、测试和文档时必须显式检查时间字段是否满足时间戳要求，避免后续需求遗忘。

## 14. 权限与合规边界

- 用户端、平台后台、代理后台使用不同认证域。
- 平台后台按 RBAC 管理全平台资源。
- 代理后台只能查看 root_agent_id = 当前代理 的团队数据。
- 代理后台不能修改用户资产、订单、行情策略、闪兑配置、新币认购或解禁规则。
- 外部行情交易对只展示 Bitget / HTX 行情，不允许后台控制外部真实行情。
- 新币策略行情仅允许用于新币或内部测试交易对。
- 高风险后台操作必须审计、二次确认并记录操作原因。

## 17. 关键约束

- 一期不做微服务，先采用模块化单体。
- 新币策略行情必须与外部真实行情隔离。
- 新币 K 线生成必须基于持久化策略、版本和检查点，不能只依赖内存定时器。
- 用户资产变动必须有流水。
- `wallet_accounts.locked` 是汇总余额，解禁必须依据锁定仓位或锁定订单。
- 固定日期解禁可聚合锁定仓位，相对周期解禁必须按订单拆分。
- MySQL 保存核心交易业务数据，MongoDB 保存行情与 K 线历史，Redis 保存实时行情缓存。
- 所有对外时间字段必须使用 Unix milliseconds 时间戳；新增接口、事件、缓存和文档不得遗忘该要求。
- 平台管理员后台与代理后台必须使用不同认证域和权限边界。
- 代理后台默认只读团队数据，不能修改用户资产、订单或行情。
- 高风险后台操作必须审计。
- 后续扩展模块不得破坏一期核心账户、订单、行情边界。
